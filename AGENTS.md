# Project Instructions

## 项目总则

- 所有内部文档、讨论、设计、规划和实现方案都用中文，并持久化保存，保持精简、可维护。
- 不特意强调时，不用建新分支，默认在 `main` 分支上操作。
- 不要生成 `.superpowers` 目录；如果出现，应删除。
- 项目处于快速迭代阶段。除非明确要求，不运行测试、构建或验证命令。
- **Agent 在执行时会被给予最高权限，无需考虑耗时和 token 消耗量，尽可能完成目标。**

## 项目定位

本项目的主线已经从“双语技术文档站”转向“面向世界的英文中国内容产品”。

公开站点的核心目标是用英文输出更好、更多、更高质量的关于中国的内容，覆盖中国经济、产业、财政、能源、科技、创新、政策和公司观察。内容来源主要由用户指定，通常是高质量中文文章、报告、研究、新闻或文章列表。Agent 的任务不是简单翻译，而是将中文信息优势转化为英文读者和 AI Search 都容易发现、理解、引用和继续阅读的原创内容。

既有技术文档 topic 继续保留，但作为存量内容和辅助流量资产，不再作为 landing page 的主要展示重心。

## 公开站点语言与 URL

- 新主线内容只做英文公开页，不再为新内容生成中文公开页。
- 新主线内容 URL 不使用 `/en/` 或 `/zh/` 前缀，优先使用简单清晰的英文路径。
- 推荐 URL 结构：`/<topic>/<stable-page-slug>/`，例如 `/china-economy/china-2026-reflation-turning-point/`。
- 旧技术文档可以继续保留现有 `/en/...` 与 `/zh/...` 路径，除非用户明确要求迁移。
- 公开 URL 一旦发布应尽量稳定。早期分类可以调整，但不要频繁改变已发布页面 URL。
- 如果必须调整分类或 URL，应使用 301 重定向、内部链接和 hub 页面承接搜索权重。
- 分类会随内容增多而演化，因此早期分类只作为导航和聚合方式，不应把页面价值绑定死在分类名上。

## 内容生产原则

用户提供来源，Agent 负责内容升级。

1. 用户提供单篇文章、文章列表、原文、截图、链接或摘要。
2. Agent 读取用户指定来源，抽取事实、观点、框架、术语、问题意识和潜在搜索需求。
3. Agent 结合站内已有内容判断分类、URL、站内链接和知识图谱关系。
4. Agent 对关键事实做必要核验，尤其是数据、时间、政策名、公司名和容易误导的判断；不需要把“寻找来源”作为主要任务。
5. Agent 将原文重写为英文原创长文，不逐段翻译，不复刻原文结构，不把公众号文章包装成转载。
6. Agent 应补充中文语境解释、结构化框架、反例、边界条件、数据源地图、术语解释、指标清单、相关概念和后续阅读路径。
7. Agent 应同时服务 SEO、AI Search 和人类深度阅读，但不能为了 SEO 牺牲文章从上到下的阅读流畅感。

## 内容质量要求

公开文章必须像真正给英文读者阅读的高质量长文，而不是文档站条目、SEO checklist 或机器摘要。

- 内容应自然、连贯、有叙事推进，适合从头到尾阅读。
- SEO 结构要隐含在标题、开头、段落组织、定义、FAQ、来源和内部链接中，不能让读者感觉在读关键词堆砌。
- 文章长度根据原文和主题自然决定。对高质量长文来源，公开英文页通常应比普通摘要更长、更完整，而不是明显短于原文的压缩版。
- 可以使用表格、定义框、指标清单、FAQ 和小结，但必须服务阅读，不要机械堆模块。
- 必须区分可核验事实、来源作者观点、Agent 综合判断和不确定情景。
- 对战争、资产价格、资金流、政策意图等强判断要谨慎表达，不能把观点写成事实。
- 不得输出投资建议、确定性预测或夸张标题。
- 文章的主要价值必须来自清楚解释、真实判断、中文信息差、结构化理解、来源透明和读者可继续追踪的线索。

## SEO 与 AI Search 要求

SEO 是新主线内容的核心交付物之一，但不是唯一目标。

- 每篇文章都要先确定英文搜索表达，而不是翻译中文标题。
- 标题、description、URL、H1、前 200 词、FAQ、定义段和内部链接都要围绕真实英文查询设计。
- 每篇文章应给出清楚的 quick answer，但不能让全文变成碎片化摘要。
- 优先覆盖英文世界缺解释、中文世界信息密度高的主题。
- 每篇文章至少提供一种英文 SERP 中稀缺的价值：中文术语映射、产业链图谱、政策时间线、关键数据表、常见误解、反例、边界条件、影响说明或可追踪数据源。
- 公开页应保留必要来源链接，尤其是用户指定来源、官方数据、政策文件、公司公告和权威资料。
- 可以使用 Article JSON-LD、清晰发布日期和修改日期、Open Graph 元数据和可提取的 FAQ。
- 不使用旧式关键词堆叠，不为了 SEO 牺牲可信度和人类阅读体验。

## 知识图谱与站内关系

新主线内容需要逐步建立类似 Obsidian 的知识图谱。

- 每篇新主线文章应维护 frontmatter 或等价结构中的 `topics`、`tags`、`entities`、`related` 等关系字段。
- 初期采用静态数据 + 前端交互图，不引入复杂后端系统。
- 页面之间要通过正文链接、相关阅读、topic hub 和知识图谱形成阅读网络。
- 分类可以演化，但实体、主题和文章之间的关系要尽量持续积累。
- 新文章发布时，Agent 应检查是否能链接到已有相关文章；如果没有，也应在内部计划中记录未来可补的节点。

## 站点与设计方向

- 设计前端UI和样式的改动，使用 Frontend Design 这个 skills
- Landing page 以新主线内容为核心，定位可采用 “Deep English explainers on China’s economy, industry, technology, and energy” 这类更适合 SEO 和 AI Search 的表达。
- 既有技术文档 topic 保留，但不在 landing page 主视觉和主要入口中直接突出。
- 整体视觉保留 retro 基础风格，不再维护多套主题。
- 移除公开站点语言切换按钮；新主线内容不需要中文版本切换。
- 主题/阅读模式切换如保留，应使用轻量、精致的控件风格，可参考 `https://uiverse.io/Galahhad/silent-robin-40`，但不要恢复多主题体系。
- 长文阅读页应优先保证正文宽度、行距、目录宽度和阅读流畅性。右侧目录不应过窄导致过早换行，也不应挤压正文形成文档站式阅读体验。
- 新主线内容可以摆脱传统文档站结构，更接近高质量专题文章、研究解读和知识网络。

## 内部产物与公开产物

本项目有两类产物：

- 内部工作产物：来源记录、选题判断、SEO 计划、页面意图、来源地图、风险判断、知识图谱关系设计和人工审核问题，主要放在 `content-plans/`。
- 公开读者产物：最终面向读者的英文文章、topic hub、知识图谱页面和保留的技术文档，放在 `src/content/docs/` 或后续确认的公开内容目录中。

公开页面不得出现我们的讨论或策划内容，不得出现“搜索意图”“页面目标”“差异化”“来源需求”“人工审核”“竞品观察”“SEO 计划”等内部字段或提纲词。

不得把大纲报告、研究记录或内部检查项直接复制到公开页面；只能把事实、判断和来源消化后，改写成读者可用的解释、叙事、例子、对比、图谱或排错路径。

## 旧技术文档保留规则

- 旧的 Astro、Hermes、Swarms、Awesome Skills 等技术文档可以继续保留。
- 修复旧文档 bug、链接、导航或样式时，沿用其原有结构即可。
- 除非用户明确要求，不要把旧技术文档作为新 landing page 的核心内容。
- 旧文档的双语结构可以暂时保留，不代表新主线内容继续双语化。

## 目录约定

- `articals/`：用户提供的原始文章或临时来源材料。保留用户原文，不要随意改写或删除。
- `content-plans/`：Agent 生成的内部研究、策划、SEO、来源和知识图谱关系记录。不要把它变成状态机、队列、GSC 仓库或人工命令手册。
- `src/content/docs/`：当前公开内容目录。后续如果为了去掉 `/en` 路径而调整目录或路由，需要先形成方案并保持重定向。
- `tools/`：Agent 内部工具。人不需要学习这套 CLI。
- `functions/`：Cloudflare Pages Functions。当前用于 topic 子域名等请求改写；除非确认不再需要相关部署路由能力，否则不要删除。

## 凭据与外部能力

- `GITHUB_TOKEN`：推荐提供，用于提高 GitHub API 额度和读取公开仓库元信息。
- 搜索 API：可选。用户指定来源时，搜索不是主要流程；需要事实核验、SERP 判断或竞品观察时再使用网页搜索或搜索 API。
- Google Search Console：发布后复盘用，不是生成文章的前置条件。
- 站点访问统计：发布后优化用，可用 Cloudflare Web Analytics 或其他统计工具。
