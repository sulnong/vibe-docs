# Project Instructions

## 项目总则

- 所有文档都用中文。
- 讨论、设计、规划、实现方案都要形成文档并持久化，保持文档精简，可维护。
- 不特意强调时，不用建新分支，默认在 `main` 分支上操作。
- 不要生成 `.superpowers` 目录；如果出现，应删除。
- 项目处于快速迭代阶段。除非明确要求，不运行测试、构建或验证命令。
- **Agent 在执行时会被给予最高权限，不要考虑耗时和token消耗量，尽可能的完成目标**

## 项目定位

这是一个 agent-assisted 的双语技术文档站。公开站点用 Astro + Starlight 承载主题优先的中英文文档；内部工具只辅助 agent 做选题研究、大纲报告和文档生成，不对公开读者展示。

本项目有两类产物：

- 内部工作产物：研究记录、选题判断、大纲报告、页面意图、来源地图和人工审核问题，主要放在 `content-plans/`。
- 公开读者产物：最终面向读者的中英文技术文档，只放在 `src/content/docs/`。

## 内容生产流程

1. 用户在 agent 中输入一个 topic。
2. agent 尝试 clone 官方仓库；如果耗时太久就放弃，改用 GitHub API、README、raw 文件和官方文档站。
3. agent 查找更多与选题相关的文档、release notes、examples、issues、FAQ、迁移指南、源码示例、社区问题、竞品页面和 SERP 结果，从尽可能多的来源获取信息。
4. agent 抽取事实并标注来源，补充例子、坑点、对比、排错路径和搜索需求。
5. agent 优先制造别人没有的内容，例如真实任务地图、反例、边界条件、版本差异、排错路径和决策清单。
6. agent 生成 `content-plans/<topic-slug>/outline.md` 选题大纲报告。
7. 用户人工审核大纲报告，确认后 agent 才进入文档生成阶段。


## 大纲报告要求

- 大纲报告是进入公开文档生成之前的最后一道审核关卡，必须清楚、具体、详实，但不可臃肿，方便人工审核者理解和评估。
- 必须包含研究流程、来源地图、SERP/竞品观察、URL 路径设计、页面组织方式和人工审核问题。
- 页面部分要详实：每个页面都要写清楚英 URL、搜索意图、要回答的问题、页面结构、来源需求和差异化内容。

## 公开文档要求

公开文档的目标很简单：生成真实、自然、符合人类阅读习惯的技术文档。

- `src/content/docs/` 只放最终面向读者的公开内容。讨论过程、策划依据、页面意图、研究记录、审核问题和 agent 工作语言只能放在 `content-plans/` 等内部产物中。
- 公开页面不得出现我们的讨论或策划内容，不得出现“搜索意图”“页面目标”“差异化”“来源需求”“人工审核”“竞品观察”等内部字段或提纲词。
- 公开页面不解释“为什么要写这页”或“如何策划这页”，只直接回答读者正在面对的问题。
- 不得把大纲报告、研究记录或内部检查项直接复制到公开页面；只能把事实、判断和来源消化后，改写成读者可用的说明、步骤、示例、对比或排错内容。
- 页面结构应自然服务于当前主题，不要所有页面套同一个模板，不要机械重复同一批小标题，可以使用代码块、命令、配置、表格、提示、排错步骤和相关页面链接，但这些元素必须有上下文、有解释、有实际作用。
- 页面数量、页面长度和文档规模根据 topic 自然决定。该长则长，该短则短，但不能只有标题堆砌、实际内容很少，参考资料列一大堆。
- 页面的主要价值必须来自清楚的解释、可操作的信息和真实读者会关心的判断。

## Topic 导航约定

- 暂时不要建设总 topic 目录页，除非用户明确确认放置位置。
- 一个 topic 应被视为一个独立文档空间,同一 topic 的中英文页面 slug 保持一致,未来可以映射到一个二级域名。
- 页面左侧导航应主要展示当前 topic 内的页面，不把所有 topic 混在一起。
- 每个 topic 的首页应承担导览页角色，帮助读者判断从哪里开始、按什么路径继续阅读。
- 同 topic 页面之间应形成阅读闭环，优先链接到当前 topic 内的安装、快速开始、配置、排错、对比和资源页面。
- 可以在低干扰位置提供“探索更多 topic”的入口，例如站点首页、页脚、topic index 页底部或资源页；这个入口只服务于感兴趣的读者，不应干扰当前 topic 的阅读路径。

## 目录约定

- `tools/`：agent 内部工具。人不需要学习这套 CLI；agent 可用 `research` 生成大纲报告，用 `draft` 生成草稿，用 `check` 做结构检查。
- `content-plans/`：agent 生成的选题大纲报告目录，允许保存内部研究、策划、判断、来源和页面设计。不要把它变成状态机、队列、GSC 仓库或人工命令手册。
- `src/content/docs/`：公开文档内容，只放最终面向读者的中英文页面，不放内部策划、创作说明或审核备注。

## 凭据与外部能力

- `GITHUB_TOKEN`：推荐提供，用于提高 GitHub API 额度和读取公开仓库元信息。
- 搜索 API：可选。没有搜索 API 时，agent 仍可用网页搜索和人工 SERP 观察；需要稳定批量化时再接 SerpAPI、DataForSEO 或 Google Custom Search JSON API。
- Google Search Console：发布后复盘用，不是生成大纲报告的前置条件。
- 站点访问统计：发布后优化用，可用 Cloudflare Web Analytics 或其他统计工具。
